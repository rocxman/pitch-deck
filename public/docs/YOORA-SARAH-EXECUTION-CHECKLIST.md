# Phase 3 — Security Remediation Implementation

## Objective

Fix confirmed vulnerabilities and establish security foundation before full ecommerce migration.

## P1 Security Checklist

### SEC-001 — Fix unpaid order detail page IDOR

- [ ] Identify route handler / page data source for `/id/profile/unpaid/<order_id>`.
- [ ] Add server-side ownership check.
- [ ] Ensure user can only access own order.
- [ ] Return `403` or generic `404` for unauthorized order.
- [ ] Add user A/B regression test.

Acceptance:

- [ ] User B cannot access user A unpaid order page.
- [ ] No order data renders for unauthorized user.
- [ ] Unauthorized access logged safely.

### SEC-002 — Fix order detail API BOLA

- [ ] Identify handler for `/api/webstore/orders/<order_id>`.
- [ ] Scope query by authenticated user.
- [ ] Remove unnecessary payment metadata from response.
- [ ] Add cross-account test.

Acceptance:

- [ ] User B cannot fetch user A order JSON.
- [ ] API returns `403`/`404`, not PII.
- [ ] Payment session IDs not exposed unnecessarily.

### SEC-003 — Fix unpaid order list exposure

- [ ] Identify list page data source `/id/profile/unpaid`.
- [ ] Filter by current user.
- [ ] Add multi-user test.

Acceptance:

- [ ] User sees only own unpaid orders.

### SEC-004 — Fix unpaid order collection API exposure

- [ ] Identify `/api/webstore/orders?status=perlu_dibayar` query.
- [ ] Scope by current user.
- [ ] Audit other status filters.

Acceptance:

- [ ] Collection returns only current user orders.
- [ ] All statuses tested.

### SEC-005 — Fix guest wishlist takeover via `X-Device-ID`

- [ ] Replace caller-controlled `X-Device-ID` authority.
- [ ] Implement server-issued guest token.
- [ ] Store securely with signing/integrity protection.
- [ ] Bind guest wishlist to trusted identity.
- [ ] Reject forged/unknown identifiers.

Acceptance:

- [ ] Session B cannot replay Session A guest state.
- [ ] GET/POST/DELETE require trusted guest binding.
- [ ] Legacy identifiers handled safely.

## P2 Security Checklist

- [ ] Audit cart for same `X-Device-ID` issue.
- [ ] Audit checkout pre-login state.
- [ ] Fix login/refresh token instability.
- [ ] Confirm auth cookie flags:
  - [ ] `Secure`
  - [ ] `HttpOnly`
  - [ ] `SameSite`
- [ ] Add account verification:
  - [ ] email verification
  - [ ] phone/OTP verification if required
- [ ] Add promo/voucher abuse prevention.
- [ ] Add rate limiting for sensitive flows.
- [ ] Review downstream renderers:
  - [ ] email
  - [ ] invoice
  - [ ] packing slip

## Acceptance Criteria

- 5 confirmed vulnerabilities fixed or explicitly scoped.
- Retest completed with 2+ accounts/sessions.

# Final Agency Execution Principles

Meanwhile should maintain these rules throughout the project:

1. Be factual and evidence-based.
2. Separate bug fix from feature request.
3. Keep third-party costs client-owned/client-paid unless written otherwise.
4. Keep source code ownership clear after full payment.
5. Use staging before production.
6. Never run destructive/security-invasive testing without written authorization.
7. Keep acceptance criteria per deliverable.
8. Document every handover item.
9. Make maintenance scope explicit.
10. Protect client trust by being transparent about risks, limitations, and responsibilities.
